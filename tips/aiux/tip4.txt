This UX idea is **very strong**: the “Now” pane becomes the agent’s live cockpit. The user asks a system question, and the pane instantly starts assembling the answer visually: ranking, probing, grouping, surfacing evidence, and preloading the obvious next drill-downs before the agent speaks.

## Core direction

Keep the left nav stable. The center “Now” pane becomes a **realtime AI workbench**.

When the agent takes charge:

The **Now** nav item shifts from green to a vivid purple “agent active” state. The pane immediately enters a live mode: cards slide in, risk rings pulse, ranked lists reshuffle, dependency lines appear, evidence trays populate, and likely follow-up actions are precomputed.

The key is this split:

**Speech is slow. UI is immediate.**

The agent should push visual updates within tens of milliseconds using cached state and speculative reads, while audio/text trails behind with the explanation.

## Top questions that should produce amazing JITUX

1. **“What is the hottest thing in the queue right now?”**
   The Now pane should instantly rank all packets by risk, freshness, blockedness, blast radius, and decision urgency. Cards fly into a “Top Heat” stack with the top one enlarged and alternatives already expanded.

2. **“Why is JPCM degraded?”**
   The pane should switch into incident mode: timeline left-to-right, impacted systems as chips, suspected causes ranked, evidence attached, and “next best action” pinned at the top.

3. **“Drill into JeRyu.”**
   The agent should replace the generic inbox with a JeRyu-specific system slice: current state, recent deltas, open decisions, incidents, dependent tools, owners, and stale evidence.

4. **“Compare JeRyu, Jekko, and Jailgun.”**
   This should create a three-lane race view: health, queue depth, blockers, incidents, throughput, decision debt, and confidence. The agent ranks them live as data arrives.

5. **“What can I safely decide right now?”**
   The pane should filter down to decisions with enough evidence, show recommended choices, risk delta, expiry, and a one-click “approve / defer / ask for evidence” action set.

6. **“What changed since I last looked?”**
   A delta stream view: new packets, rank changes, resolved blockers, promoted memories, regressions, and suspicious changes. Cards should animate from “was” to “now.”

7. **“What is blocking throughput?”**
   The pane should generate a bottleneck graph: blocked adapters, missing authority, stale leases, unverified incidents, slow approvals, and repeated failed paths.

8. **“Show me the evidence behind AP-88.”**
   Expand the card into an evidence stack: source events, traces, logs, screenshots, adapter state, confidence, contradicting evidence, and freshness.

9. **“What should the system do next without me?”**
   The pane becomes an autonomy board: safe automatic actions, actions needing explicit approval, risky actions, and already-prefetched rollback paths.

10. **“Is this incident related to earlier failures?”**
    The agent builds a relationship graph: similar incidents, shared adapters, repeated error signatures, common owners, previous remediations, and policy/memory matches.

## The killer UX model: JITUX packets

Everything the agent does should stream as structured UI patches, not as prose. The frontend should not wait for a full answer. It should receive tiny, composable events like:

```ts
type JituxEvent =
  | { type: "agent_mode"; mode: "taking_charge" | "observing" | "awaiting_user"; accent: "purple" | "green" }
  | { type: "pane.reset"; pane: "now"; title: string; subtitle?: string }
  | { type: "metric.upsert"; id: string; label: string; value: string | number; severity?: "low" | "medium" | "high" }
  | { type: "card.upsert"; card: JituxCard }
  | { type: "card.rank"; ids: string[]; reason: string }
  | { type: "lane.upsert"; lane: JituxLane }
  | { type: "graph.upsert"; graph: JituxGraph }
  | { type: "timeline.upsert"; timeline: JituxTimeline }
  | { type: "evidence.attach"; cardId: string; evidence: EvidenceItem[] }
  | { type: "action.attach"; cardId: string; actions: JituxAction[] }
  | { type: "prefetch.ready"; target: string; payloadRef: string }
  | { type: "voice.cue"; text: string };
```

A card should be designed for partial truth:

```ts
type JituxCard = {
  id: string;
  kind: "incident" | "decision" | "system" | "tool" | "memory" | "approval" | "blocker";
  title: string;
  subtitle?: string;
  status: "probing" | "partial" | "verified" | "stale" | "blocked";
  severity?: "low" | "medium" | "high" | "critical";
  rank?: number;
  whyNow?: string;
  confidence: number;
  freshnessMs?: number;
  riskDelta?: {
    from: "low" | "medium" | "high" | "critical";
    to: "low" | "medium" | "high" | "critical";
    reason: string;
  };
  recommended?: {
    label: string;
    rationale: string;
  };
  alternatives?: Array<{
    label: string;
    tradeoff: string;
  }>;
  actions?: JituxAction[];
  drilldowns?: Array<{
    label: string;
    query: string;
    prefetchKey?: string;
  }>;
  evidenceRefs?: string[];
  expiresAt?: string;
};
```

The important part: **cards can appear as “probing” immediately**, then become verified as data lands. That makes the UI feel alive without pretending certainty.

## Pane modes

The Now pane should have multiple live compositions, all driven by the same event stream.

**Inbox mode**
Decision packets, urgent items, blockers, memory promotions.

**Incident mode**
Timeline, blast radius, suspected causes, live evidence, rollback/action rail.

**System drill-down mode**
One system/tool in focus: JeRyu, Jekko, Jailgun, adapter, queue, memory, approvals.

**Comparison mode**
Multiple systems side by side with ranked health, queue debt, blockers, throughput.

**Evidence mode**
A compact forensic view: traces, logs, source packets, related decisions, stale/conflicting evidence.

**Autonomy mode**
What the agent can do now, what needs approval, what it already staged.

## “AI taking charge” interaction

The flow should feel like this:

**0–50ms**
Now button turns purple. Pane title changes. Skeleton lanes appear. Agent status chip says something like “Taking charge.”

**50–150ms**
Cached queue snapshot paints: top packets, metrics, stale risk, current degraded systems.

**150–400ms**
Speculative probes start filling cards: JeRyu/Jekko/Jailgun lanes, incident candidates, blockers, obvious drilldowns.

**400–900ms**
Ranked cards settle. Evidence attaches. The top recommendation gets pinned.

**900ms+**
Voice/text starts explaining what the user is already seeing.

This creates the magic: the user asks a question and the system starts visibly working **before** the agent talks.

## Backend architecture

Use Rust as the realtime truth engine and TypeScript/React as the visual stage.

```txt
jmcp/
  crates/
    jmcp-core/             domain models, packet ranking, risk scoring
    jmcp-server/           axum API, websocket/SSE, auth, sessions
    jmcp-agent-runtime/    agent runs, planning, tool orchestration
    jmcp-adapters/         JeRyu/Jekko/Jailgun/backbone adapters
    jmcp-eventlog/         append-only event store
    jmcp-ranking/          queue ranking and urgency scoring
    jmcp-jitux/            UI patch schemas and reducers
  apps/
    web/                   Vite + React + TypeScript
```

The server should expose two streams:

```txt
/ws/jitux/:session_id       fast UI events
/ws/voice/:session_id       transcript/audio/status
```

Do not tie UI rendering to final model output. The agent runtime should emit JITUX as it reasons.

Internally:

```txt
User query
  -> Intent classifier
  -> Snapshot fetch from hot cache
  -> JITUX planner emits immediate pane reset + skeleton + cached cards
  -> Tool fanout starts in parallel
  -> Ranker updates cards as results arrive
  -> Evidence resolver attaches proof
  -> Action planner precomputes next likely actions
  -> Voice/text summarizer explains the settled view
```

## Frontend architecture

React should treat the Now pane as a **realtime reducer over JITUX events**.

```ts
const state = jituxReducer(previousState, incomingEvent);
```

Recommended pieces:

```txt
Vite
React
TypeScript
Zustand or Jotai for pane state
TanStack Query for slower background data
WebSocket for JITUX event stream
Framer Motion / Motion for layout transitions
React Window / Virtuoso for huge queues
Zod for event validation
```

Motion rules:

Cards should use FLIP/layout animation so rank changes are smooth. New cards should enter fast from the relevant edge, not randomly. Risk changes should pulse once, not loop forever. Evidence should “snap attach” to cards. Preloaded drilldowns should feel instant when clicked or spoken.

## Ranking model

Every packet should get a visible reason, not just a score.

```ts
type RankReason = {
  score: number;
  factors: {
    severity: number;
    freshness: number;
    blastRadius: number;
    blockedness: number;
    decisionReadiness: number;
    evidenceCompleteness: number;
    expiryPressure: number;
    repeatIncidentLikelihood: number;
  };
  explanation: string;
};
```

This lets the UI animate honestly: “AP-88 moved above AP-89 because write exposure risk increased and evidence remains missing.”

## Best implementation principle

Do not build fixed pages for every system.

Build a **JITUX grammar**:

```txt
metrics
cards
lanes
ranked lists
timelines
graphs
evidence trays
action rails
comparison tables
status toasts
```

Then let the agent compose those blocks dynamically. That makes the interface infinitely extensible without requiring a custom UI for every future tool.
