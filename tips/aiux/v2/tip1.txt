The coolest version is a **3D Command Deck**: the Now pane becomes a living stack of “prepared views,” not a static page. The agent is constantly building, ranking, and warming panes in the background. The user sees 5–20 high-level previews in a fast 3D stack, and the active pane snaps forward only after it is already hydrated.

The key design principle:

**Never animate the agent’s uncertainty as slow loading. Animate the agent’s control as instant reorganization.**

## The navigation concept: “Live Deck”

The right side of the Now pane becomes a 3D stack of panes:

```txt
[ Active Focus Pane ]     [ tilted stacked previews ]
                           JeRyu degraded
                           AP-88 evidence gap
                           Jekko throughput
                           Jailgun authority
                           Memory promotions
                           Replay incidents
                           Queue heat
```

Each preview is a miniature system pane with just enough information:

```txt
JeRyu
high risk · 3 blockers · 1 action ready
evidence 71% · last update 4s ago
```

The stack constantly reorders itself as the agent learns more. The top card is “most likely the user cares.” The bottom cards are still alive, still being updated, just lower priority.

When the user says “drill into JeRyu,” the JeRyu preview does **not load**. It has already been incubated. It flies forward, expands, and becomes the active pane.

## The best visual pattern

I would build three layers:

### 1. Active pane

This is the real React/DOM pane. It contains full cards, buttons, evidence, drilldowns, actions, and text.

Only **one** pane should be fully active.

### 2. Warm panes

Keep 2–4 likely next panes pre-rendered but hidden/offscreen. These are ready to become active instantly.

Examples:

```txt
JeRyu drilldown
AP-88 evidence
Compare JeRyu/Jekko/Jailgun
Autonomy actions
```

### 3. Preview deck

The 5–20 card stack should be lightweight. Do **not** render 20 full React panes. Render preview models only: title, risk, counters, confidence, small sparkline, status chips.

This is what makes it feel insane-fast.

([vitejs][1])

The 3D layer should be **React Three Fiber + Three.js** for the deck, depth, lighting, parallax, connector lines, and background “agent organizing” visuals. React Three Fiber is a React renderer for Three.js, so it lets you compose interactive 3D scenes as React components while still using the React ecosystem. ([Poimandres Documentation][2])

This gets you the cool 3D feel without sacrificing text clarity, accessibility, or interaction speed.

## The killer behavior: Pane Incubation

The agent should build panes before showing them.

```ts
type PaneStatus =
  | "predicted"
  | "incubating"
  | "warm"
  | "active"
  | "stale"
  | "discarded";
```

A pane starts as predicted, becomes incubating while data loads, becomes warm once it has enough structure, then can become active instantly.

```ts
type PaneVM = {
  id: string;
  kind:
    | "queue"
    | "incident"
    | "system"
    | "comparison"
    | "evidence"
    | "approval"
    | "autonomy";

  title: string;
  subtitle?: string;

  rank: number;
  focusScore: number;
  confidence: number;
  severity: "low" | "medium" | "high" | "critical";

  status: PaneStatus;

  preview: {
    headline: string;
    chips: string[];
    counters: Array<{ label: string; value: number | string }>;
    sparkline?: number[];
  };

  cards: JituxCard[];
  actions: JituxAction[];
  evidenceRefs: string[];

  predictedNext: string[];
  updatedAt: string;
  expiresAt?: string;
};
```

The active pane is just one `PaneVM`. The stack is an ordered list of other `PaneVM`s.

## JITUX event protocol

The backend should stream UI intent as tiny events. Not full pages. Not prose. UI patches.

```ts
type DeckEvent =
  | {
      type: "agent.mode";
      mode: "observing" | "taking_charge" | "awaiting_user";
      accent: "green" | "purple";
    }
  | {
      type: "deck.reset";
      title: string;
      subtitle?: string;
    }
  | {
      type: "pane.predict";
      pane: PaneVM;
    }
  | {
      type: "pane.patch";
      paneId: string;
      patch: Partial<PaneVM>;
    }
  | {
      type: "pane.warm";
      paneId: string;
      readyLevel: "preview" | "summary" | "full";
    }
  | {
      type: "pane.activate";
      paneId: string;
      transition: "fly-forward" | "explode" | "swap" | "focus-tunnel";
    }
  | {
      type: "deck.rank";
      orderedPaneIds: string[];
      reason: string;
    }
  | {
      type: "card.patch";
      paneId: string;
      cardId: string;
      patch: Partial<JituxCard>;
    }
  | {
      type: "evidence.attach";
      paneId: string;
      cardId: string;
      evidence: EvidenceItem[];
    };
```

This lets the agent update the interface at reasoning speed.

## Rust backend stack

Use Rust as the realtime control layer:


For a single-node MVP, WebSocket events can come directly from `tokio` channels.

For production, use **NATS JetStream** as the event backbone. It supports streaming/replay behavior, and its docs describe replay policies that can deliver stored messages instantly or at original publish rate. That is perfect for restoring a user’s Now pane, replaying what the agent saw, or rehydrating another client. ([NATS Docs][3]) JetStream also has acknowledgment-based delivery semantics, which matters once these UI events become action-bearing system events. ([NATS Docs][3])

The backend should have these services:

## Performance model

The trick is to target different speeds for different layers:

```txt
0–16ms:
  Now turns purple
  cached deck appears

16–80ms:
  preview panes rank
  skeleton active pane appears

80–250ms:
  warm panes hydrate
  top cards get evidence counters

250–600ms:
  active pane becomes detailed
  actions attach
  evidence expands

600ms+:
  agent voice explains what the user already sees
```

Do not wait for the model. Do not wait for all tools. Do not wait for complete evidence.

Render immediately from the last known snapshot, then refine.

## Frontend performance rules

This is the part that makes it actually feel instant:

```txt
Only one full active pane.
Only 2–4 warm panes.
All other panes are previews.
All animations use transform + opacity.
No layout animation on width/height/top/left.
No expensive shadows on 20 moving elements.
Batch socket events per animation frame.
Never rank in React render.
Never fetch from a component during animation.
Never mount huge evidence trees until active.
```

Use Zustand for hot animation state because you can update tiny slices without rerendering the whole app. Use TanStack Query for normal data fetching, not for the high-frequency deck stream.

The deck reducer should look like this:

```ts
function applyDeckEvent(state: DeckState, event: DeckEvent): DeckState {
  switch (event.type) {
    case "pane.predict":
      return upsertPane(state, event.pane);

    case "pane.patch":
      return patchPane(state, event.paneId, event.patch);

    case "deck.rank":
      return {
        ...state,
        orderedPaneIds: event.orderedPaneIds,
        lastRankReason: event.reason,
      };

    case "pane.activate":
      return activatePane(state, event.paneId, event.transition);

    default:
      return state;
  }
}
```

Then render order is just:

```ts
const panes = orderedPaneIds.map((id, index) => {
  return {
    pane: paneById[id],
    transform: getDeckTransform(index, id === activePaneId),
  };
});
```

## 3D transform model

The deck should feel physical and legible:

```ts
function getDeckTransform(index: number, active: boolean) {
  if (active) {
    return {
      x: 0,
      y: 0,
      z: 80,
      rotateY: 0,
      rotateX: 0,
      scale: 1,
      opacity: 1,
    };
  }

  return {
    x: 720 + index * 18,
    y: 80 + index * 22,
    z: -index * 38,
    rotateY: -18,
    rotateX: 2,
    scale: 1 - index * 0.025,
    opacity: Math.max(0.35, 1 - index * 0.06),
  };
}
```

A card moving from rank 12 to rank 1 should not “teleport.” It should rocket forward, overshoot slightly, then settle. That communicates that the agent discovered something important.

## The smartest interaction: focus without navigation

The user should not feel like they are changing pages. They are changing **focus**.

Examples:

```txt
“What’s hottest?”

  Stack: systems, evidence, incidents, approvals

“Drill into JeRyu.”
  JeRyu preview flies forward
  AP-88 and Jekko stay visible behind it

“Compare JeRyu and Jailgun.”
  Stack explodes into two lanes
  Shared blockers appear between them

  Evidence cards fly into the active pane
  Unsupported claims fade down
```

This makes the Now pane feel like a live command surface rather than a dashboard.
