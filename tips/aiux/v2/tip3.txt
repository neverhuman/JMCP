Yes — the coolest version is a **3D “Mission Deck”**: the Now pane becomes a fast, AI-controlled stack of live panes/cards, almost like a spread of tiny VMs. The agent is constantly preparing, ranking, and swapping panes before the user asks for them.

I’d build it like this.

## The UX concept: “Gravity Deck”

The right/main pane becomes a 3D stack:

```txt
              [ active pane: queue risk ]
        [ jeryu trace ] [ jailgun lane ] [ replay ]
   [ approvals ] [ evidence ] [ memory promotion ] [ adapter health ]
```

The **active pane** is large and readable in the center. Behind it, 5–20 small preview panes sit in perspective, angled like a physical card stack or command-center wall. They are not static thumbnails; they are live previews with tiny counters, pulsing edges, mini sparklines, and status chips.

When the agent starts investigating:

1. `Now` turns electric purple.
2. The deck appears immediately with skeleton panes.
3. Data panes fly in from the right/background.
4. The most relevant pane rises to the front.
5. Other panes reshuffle in 3D depth order.
6. Hidden panes continue loading and preparing.
7. When the user says “drill into Jeryu,” that pane is already warm and snaps forward instantly.

This should feel less like navigation and more like **the AI is physically organizing the control room in real time**.

## The main rule

Do **not** render everything as WebGL.

For this kind of UI, the fastest and sharpest approach is:

**CSS 3D transforms for real cards + optional WebGL/Canvas background.**

The cards contain real text, buttons, chips, and accessibility semantics. WebGL is great for particles, depth haze, connector lines, and ambient motion, but rendering the actual cards into textures would make text, interaction, hit testing, and accessibility worse.

So the ideal stack is:

```txt
React DOM cards
CSS transform: translate3d / rotateX / rotateY / scale
GPU compositing
Canvas/WebGL background only for atmosphere + graph traces
Rust WebSocket stream feeding tiny patches
```

## Recommended technical stack

### Backend

Use Rust as the live projection and streaming authority:

```txt
Rust
 ├─ tokio
 ├─ axum
 ├─ WebSocket JITUX stream
 ├─ SQLite WAL event store
 ├─ in-memory projection cache
 ├─ bounded fanout workers
 ├─ adapter probes: Jeryu / Jekko / Jailgun / Jankurai
 └─ action authority layer
```

Tokio is the right async foundation here because it is explicitly built for high-speed Rust network applications, with async I/O, scheduling, timers, synchronization, and a multithreaded work-stealing scheduler. ([Tokio][1]) Axum is a good fit for the HTTP/WebSocket layer; its WebSocket extractor supports upgrade handlers and concurrent read/write patterns. ([Docs.rs][2])

### Frontend

```txt
Vite
React
TypeScript
CSS 3D transforms
Custom external animation store
requestAnimationFrame scheduler
Zustand or useSyncExternalStore
Optional: Motion for higher-level transitions
Optional: Pixi/WebGL/Three only for background effects
```

Vite is a strong fit because it is designed for modern frontend development with fast HMR and optimized production builds. ([vitejs][3]) React’s `useTransition` is useful for background UI updates, while `memo` helps avoid rerendering unchanged card components. ([React][4])

But the important part: **React should not drive every animation frame.** React renders the card structure. A lightweight animation engine mutates transforms on refs.

## The winning UI pattern: prepared panes, then instant commit

The agent should not say, “loading Jeryu…”

It should emit:

```json
{
  "kind": "pane.prepare",
  "pane_id": "jeryu.ecosystem",
  "priority": 91,
  "visibility": "background",
  "expected_focus": true
}
```

Then the frontend quietly builds the pane in the back of the deck. When the user asks for it, or when the agent realizes it is the most relevant, the server sends:

```json
{
  "kind": "pane.commit",
  "pane_id": "jeryu.ecosystem",
  "motion": "snap_to_front",
  "reason": "highest blast-radius relevance"
}
```

That gives you the magic: **the pane feels instant because it was already prepared.**

## JITUX frame model

I’d define the stream around four key frame types:

```ts
type JituxFrame =
  | DeckFrame
  | PaneFrame
  | CardFrame
  | FocusFrame
  | ActionFrame;
```

Example:

```ts
type DeckFrame = {
  kind: "deck.patch";
  session_id: string;
  seq: number;
  mode: "idle" | "agent_takeover" | "investigating" | "acting" | "needs_user";
  accent: "green" | "purple" | "amber" | "red";
  title: string;
  subtitle?: string;
};

type PaneFrame = {
  kind: "pane.upsert";
  pane_id: string;
  title: string;
  type:
    | "queue"
    | "jeryu"
    | "jailgun"
    | "jekko"
    | "evidence"
    | "replay"
    | "approval"
    | "adapter_health"
    | "memory";
  priority: number;
  depth: number;
  status: "preparing" | "warm" | "active" | "stale" | "blocked";
  preview: {
    headline: string;
    count?: number;
    risk?: "low" | "medium" | "high" | "critical";
    chips?: string[];
  };
  payload_ref?: string;
};
```

Then focus is its own event:

```ts
type FocusFrame = {
  kind: "focus.change";
  pane_id: string;
  reason:
    | "user_asked"
    | "agent_ranked"
    | "risk_spike"
    | "approval_expiring"
    | "adapter_degraded";
  motion: "rise" | "snap" | "fan" | "flip" | "collapse";
};
```

This lets the deck reorder without destroying the panes.

## The 3D layout

I’d use a deterministic layout function.

```ts
function getDeckTransform(index: number, activeIndex: number) {
  const d = index - activeIndex;

  return {
    x: d === 0 ? 0 : 280 + Math.abs(d) * 18,
    y: d * 22,
    z: -Math.abs(d) * 90,
    rotateY: d === 0 ? 0 : -18,
    rotateX: d === 0 ? 0 : 4,
    scale: d === 0 ? 1 : Math.max(0.72, 0.94 - Math.abs(d) * 0.035),
    opacity: Math.max(0.35, 1 - Math.abs(d) * 0.08)
  };
}
```

The active card is readable. The rest are live previews. When priority changes, cards animate to their new transforms using FLIP:

1. Measure old position.
2. Apply new order.
3. Invert transform.
4. Animate to identity.

That gives extremely smooth “cards reorganizing themselves” motion.

## Rendering strategy for speed

The fastest architecture is:

```txt
WebSocket frame arrives
  ↓
External store mutates normalized pane/card state
  ↓
Animation scheduler batches changes per animation frame
  ↓
Only affected card refs get transform updates
  ↓
React rerenders only when semantic card content changes
```

Do not rerender the whole Now pane every time a score changes. Scores, depth, transform, glow, and preview position should be cheap imperative updates.

Use React for:

```txt
card creation
content updates
buttons
accessibility
layout shell
```

Use the animation layer for:

```txt
x/y/z position
scale
rotation
opacity
glow
rank shuffling
focus transitions
```

## The coolest interaction details

The pane stack should have **modes**:

### 1. Stack mode

Default. Active pane front and center. Other panes are visible behind it.

Best for: “what’s going on?”

### 2. Fan mode

Cards fan out horizontally in 3D.

Best for: “show me everything you’re looking at.”

### 3. Tunnel mode

The active investigation becomes a depth path: queue → adapter → evidence → replay → action.

Best for: “why is this happening?”

### 4. Command mode

Actions appear as physical cards on the right edge: `prepare packet`, `replay`, `quarantine`, `request approval`.

Best for: “what should we do?”

### 5. Graph overlay mode

A transparent dependency graph floats behind the deck. Panes connect to systems: Jeryu, Jekko, Jailgun, adapters, approvals, memory.

Best for: “drill into Jeryu.”

## The important implementation trick

Use a **two-layer pane model**:

```ts
type PaneRuntime = {
  pane_id: string;
  shell: "mounted" | "unmounted";
  data: "empty" | "preview" | "warm" | "full";
  visibility: "background" | "preview" | "active";
};
```

Most panes are not full React trees. They are preview shells. Only the active pane and 2–3 likely next panes are full/warm.

So with 20 available panes:

```txt
1 active full pane
3 warm panes
16 lightweight preview panes
```

That keeps it screaming fast.
