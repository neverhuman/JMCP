Absolutely. The right move is: **do not make the agent “navigate the app.” Make the Now pane become the agent’s live cockpit.** It should feel like the system is waking up and organizing itself in front of the user before the agent finishes its first sentence.

JMCP is already the right substrate for this: the repo describes a local V1 runtime with a **Rust authority API, SQLite event store, CLI, Rust TUI, React cockpit, Telegram intake/approvals, and adapter boundaries for Jankurai, Jeryu, Jailgun, and Jekko**. The cockpit is explicitly the primary dashboard for work orders, approvals, evidence, systems, replay, and the Jeryu ecosystem graph. ([GitHub][1]) The design below keeps that architecture and adds a fast, AI-driven **JITUX stream** on top.

## Core concept: Now becomes “Agent Takeover Mode”

When the user asks anything system-related, the left rail stays fixed, and **Now turns purple** to indicate the agent is taking charge. The main pane immediately morphs into a live mission board:

**Phase 0 — instant visual response, under 100 ms**
The Now button flips purple. The pane title changes from `Now` to something like `Investigating queue`. Empty animated slots appear: `Scanning`, `Ranking`, `Evidence`, `Likely actions`.

**Phase 1 — system facts arrive before speech**
The backend streams projection snapshots: queue items, work orders, approvals, degraded systems, evidence gaps, adapter health, replay deltas. The user sees cards fly in, reorder, collapse, and glow.

**Phase 2 — agent intent narrows the view**
As the transcript becomes clearer, the Now pane swaps from general system scan to a focused board: `JPCM stream degraded`, `Jeryu blast radius`, `Jailgun run queue`, `Approval bottleneck`, etc.

**Phase 3 — pre-done next actions appear**
The agent should not merely show data. It should pre-stage the most likely next moves: `drill into evidence`, `replay last 10m`, `prepare approval`, `open adapter trace`, `compare alternatives`, `submit safe full-auto action`.

The crucial rule: **show observable work, not hidden reasoning.** The stream should expose what the agent is looking at, what tools/projections it queried, which cards were ranked, what evidence exists, and which actions are safe. It should not stream private chain-of-thought.

## The UX should feel like this

User says: “What’s going on with the queue?”

Before audio response starts, the Now pane does this:

`Now` button turns purple → top bar says `Agent taking charge` → queue cards fly in → blocked items rise to the top → degraded systems pulse → approvals get an expiry ring → evidence gaps get red edge markers → likely actions appear on the right.

Then the agent says: “I’m seeing two decision packets, one blocked adapter path, and one urgent approval expiry. I’ve already opened the replay window and ranked the safe actions.”

That is the magic. The UI proves the agent is working.

## The protocol: JITUX frames

This should be a first-class streaming contract, not ad hoc UI state. The agent/backend emits compact frames; React renders them immediately.

```ts
type JituxFrame =
  | JituxSessionFrame
  | JituxPaneFrame
  | JituxCardFrame
  | JituxRankFrame
  | JituxTraceFrame
  | JituxActionFrame
  | JituxPrefetchFrame;

type JituxBase = {
  session_id: string;
  seq: number;
  ts_ms: number;
  ttl_ms?: number;
  source: "agent" | "projection" | "adapter" | "replay" | "approval" | "frontend";
  confidence?: number;
};

type JituxPaneFrame = JituxBase & {
  kind: "pane.patch";
  pane: "now";
  mode: "observe" | "take_charge" | "drilldown" | "approval" | "incident" | "replay";
  accent: "normal" | "purple_takeover" | "risk_red" | "safe_green";
  title: string;
  subtitle?: string;
  layout: "ranked_board" | "incident_trace" | "ecosystem_graph" | "decision_inbox" | "adapter_lane";
};

type JituxCardFrame = JituxBase & {
  kind: "card.upsert";
  card: {
    id: string;
    type:
      | "work_order"
      | "approval"
      | "evidence_gap"
      | "adapter_health"
      | "risk"
      | "replay_event"
      | "autonomous_action"
      | "memory_promotion";
    title: string;
    status: "new" | "scanning" | "ranked" | "blocked" | "ready" | "needs_user" | "done";
    priority: number;
    risk: "low" | "medium" | "high" | "critical";
    summary: string;
    evidence_refs?: string[];
    related?: string[];
    drilldowns?: JituxDrilldown[];
    actions?: JituxPreparedAction[];
  };
  animation?: "fly_in" | "rise" | "pulse" | "collapse" | "replace" | "pin";
};

type JituxPreparedAction = {
  id: string;
  label: string;
  command: string;
  safety: "read_only" | "bounded_auto" | "approval_required" | "manual_only";
  ready: boolean;
  reason: string;
  preview?: unknown;
};
```

## Backend architecture: Rust owns truth, UI owns motion

Keep Rust as the authority layer. The README already positions JMCP around a Rust API and event store, with the cockpit started through Vite against the local API. ([GitHub][1]) Add this:

```txt
jmcpd
 ├─ existing authority API
 ├─ existing SQLite event store
 ├─ projections: work_orders, approvals, evidence, replay, ecosystem, adapters
 ├─ NEW jitux broker
 │   ├─ /jitux/session/open
 │   ├─ /jitux/session/:id/ws
 │   ├─ /jitux/session/:id/prefetch
 │   ├─ /jitux/session/:id/action/:action_id/prepare
 │   └─ /jitux/session/:id/action/:action_id/commit
 └─ adapter fanout
     ├─ Jeryu
     ├─ Jekko
     ├─ Jailgun
     └─ Jankurai
```

Use **WebSocket** for the main JITUX stream. Use small JSON frames, not huge payloads. Keep frame size tiny, target append-only events, and let React reconcile by ID.

The backend should have two engines:

**Projection engine:** deterministic, fast, no LLM needed. It ranks current system facts immediately.

**Agent engine:** slower but smarter. It interprets the user’s intent, writes summaries, prepares actions, and emits patches into the same stream.

That separation is what makes it feel instant. The UI should not wait for the model.

## Frontend architecture: React motion board, not page navigation

In `@jmcp/cockpit`, add a JITUX runtime:

```txt
apps/cockpit/src/jitux/
 ├─ client.ts              // WebSocket client
 ├─ store.ts               // Zustand or reducer-based frame store
 ├─ types.ts               // generated from schemas/jitux.schema.json
 ├─ ranker.ts              // local tie-breaks and optimistic ordering
 ├─ components/
 │   ├─ NowTakeoverPane.tsx
 │   ├─ LiveCard.tsx
 │   ├─ EvidenceRibbon.tsx
 │   ├─ ActionRail.tsx
 │   ├─ DrillStack.tsx
 │   ├─ ReplayStrip.tsx
 │   └─ AdapterPulse.tsx
 └─ motion/
     ├─ flip.ts
     ├─ scheduler.ts
     └─ reducedMotion.ts
```

Use transforms, opacity, and FLIP animations. Avoid layout thrash. The UI should animate card ranking by position changes, not remounting entire panes.

The left nav remains stable. Only the active Now surface mutates.

## Safety model

Do not let the UI become a fake authority. JMCP already has backend-owned approvals and persisted records across REST, CLI, cockpit, and Telegram. ([GitHub][1]) JITUX should only prepare or visualize actions unless the backend says the action is allowed.

Action classes:

```ts
type SafetyClass =
  | "read_only"          // inspect, replay, rank, summarize
  | "bounded_auto"       // capped autonomous action
  | "approval_required"  // user/token approval needed
  | "manual_only";       // show instructions only
```

The UI can make read-only and prepared states feel instant. Commits still go through JMCP authority, leases, evidence recording, replay, approvals, and adapter health tracking.

## Performance targets

For this to feel insanely alive:

| Target                     |                           Budget |
| -------------------------- | -------------------------------: |
| Purple takeover visual     | < 50 ms after partial transcript |
| First skeleton board       |                         < 100 ms |
| First projection cards     |                         < 250 ms |
| First ranking reorder      |                         < 400 ms |
| First prefetched drilldown |                         < 800 ms |
| Agent spoken summary       |                can lag behind UI |
| Action preview             |                          < 1.5 s |

Implementation details:

Use Rust `tokio::select!` / `JoinSet` to fan out projection reads. Stream partial frames as soon as each source returns. Avoid waiting for all adapters. Cache hot projections in memory. Emit stale-but-labeled snapshots instantly, then patch with fresh data.

On the frontend, batch frames per animation frame, not per WebSocket message. Keep card components memoized. Use virtualization once lists exceed ~50 cards.
